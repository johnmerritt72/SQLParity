# Changelog

## [1.3.4] — 2026-05-18

### Fixed
- **Collation false-positives on string columns.** Tables with `varchar` / `nvarchar` / `char` / `nchar` columns no longer flag as MODIFY RISKY when the live DB column has an explicit `COLLATE` clause but the folder-side `.sql` file omits it (a common pattern when the file was hand-edited or generated by a tool that strips collation). `sys.columns.collation_name` is always populated by SQL Server — for explicit values *and* for columns that inherit the database default — while the ScriptDom parser only sees what's written in the CREATE TABLE. Comparison now treats null on either side as "unspecified, inherit default" and only flags collation when both sides specify explicit values that disagree.

## [1.3.3] — 2026-05-18

### Fixed
- **Spurious MODIFY flag on tables with numeric / date columns.** The fix in 1.3.2 only covered the catalog-vs-parser gap on `MaxLength`. `sys.columns.precision` and `sys.columns.scale` report the same kind of implementation-defined non-zero values for types you can't parameterize in T-SQL — `int`=10, `bigint`=19, `tinyint`=3, `datetime`=23/3, `bit`=1, `float`=53, etc. — but the folder parser leaves them at 0. Every such column compared as modified on every run, so tables of mostly numeric columns kept flipping to MODIFY RISKY with no visible DDL drift. `Precision` is now compared only on `decimal` / `numeric`. `Scale` is compared only on `decimal` / `numeric` / `datetime2` / `time` / `datetimeoffset` — the types whose T-SQL syntax actually exposes those parameters.

## [1.3.2] — 2026-05-15

### Fixed
- **Spurious MODIFY flag from `MaxLength` mismatch on fixed-width columns.** `sys.columns.max_length` reports byte widths the T-SQL grammar can't accept — `bigint`=8, `int`=4, `smallint`=2, `tinyint`=1, `datetime`=8, `float`=8, `bit`=1. The folder-side parser leaves `MaxLength` at 0 for those types because there's no parameter to extract. Comparison now restricts the `MaxLength` check to types where the width is user-specified: `varchar` / `nvarchar` / `char` / `nchar` / `binary` / `varbinary`.
- **Auto-generated PK / FK / CK names appearing as add/drop pairs in the table detail tree.** SQL Server names anonymous constraints with shapes like `PK__AlcoholD__3214EC272833E366`. The catalog reports that name while a folder-side inline `PRIMARY KEY CLUSTERED ([Id] ASC)` parses with no name. The Indexes / Foreign Keys / Check Constraints groups in the detail tree previously matched by raw name, so the same constraint rendered as a phantom orphan-add on the DB side and an orphan-drop on the folder side. Matching now falls back to a structural fingerprint when the name follows the SMO auto-generation pattern: same shape → matched row; different shapes → distinct rows (so multiple anonymous-by-convention constraints on one table don't collide).
- **`PRIMARY KEY` columns spuriously flagged for nullability drift.** In T-SQL, any column in a PRIMARY KEY is implicitly NOT NULL — but ScriptDom doesn't apply that rule when reporting the parsed column's nullability. The catalog side correctly reports such columns as NOT NULL. The folder parser now applies the same implicit rule, so columns like `Id int PRIMARY KEY` (no explicit `NOT NULL`) compare cleanly against the live DB.

### Internal
- Auto-generated constraint name detection moved out of `CreateTableGenerator` into a shared `AutoGeneratedConstraintName` utility so the canonical DDL renderer and the table detail tree builder use the same definition.
