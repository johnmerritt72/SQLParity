# What's New in SQLParity 1.4.1

This release adds **permission comparison** — SQLParity now compares object- and schema-level permissions between two databases, classifies the risk of each change, scripts them, and surfaces them throughout the UI.

## ✨ Highlights

### 🔑 Compare object & schema permissions
SQLParity now reads and compares grants alongside object definitions:

- **Object-level** grants on tables, views, stored procedures, functions, sequences, and user-defined types — e.g. `GRANT EXECUTE ON [dbo].[MyProc] TO [AppRole]`.
- **Schema-level** grants — e.g. `GRANT EXECUTE ON SCHEMA::[sales] TO [AppRole]`.
- All three states: **GRANT**, **GRANT WITH GRANT OPTION**, and **DENY**.

It's **on by default** for database-to-database comparisons — toggle it with the new **"Permissions"** checkbox in the object-type filter. An object whose body is identical but whose grants differ now shows as **Modified**, so a permission-only change is never missed.

### 👀 See exactly what changed
- A **🔑 permissions** tag appears next to any object with grant changes in the change list — easy to spot even on objects that *also* have body changes.
- The detail pane shows a **Permissions** group listing each change, e.g. `EXECUTE → [AppRole]: none ⇒ GRANT`, color-coded by risk — for **every** object type, not just tables.

### ⚖️ Risk-aware, safety-first
Permission changes flow through the same risk tiers and destructive-change gauntlet as everything else:

- **Granting** access → Caution
- **WITH GRANT OPTION** / **DENY** → Risky
- **Revoking** access → **Destructive** (it can break a running app), so it's gated like a dropped column.

### 🧾 Synced in your script
The generated sync script gets a dedicated **Permissions** section that emits `GRANT` / `DENY` / `REVOKE` *after* the target objects exist. Each principal is guarded — if it doesn't exist on the destination, the script **fails loudly** (a `THROW` aborts the batch) rather than silently skipping the grant.

## 📌 Good to know
- **Live-vs-live only.** Permission comparison runs when both sides are databases; it's skipped (and the checkbox disabled) in folder / source-control mode.
- **Out of scope for now:** column-level grants, database/server-level permissions, and role membership.
- Your login needs permission to read `sys.database_permissions` on both databases (db_owner / sysadmin is fine).

## 🛠️ Under the hood
- The build now verifies the version baked into the packaged `.vsix` matches `version.txt` and fails on a mismatch — preventing an incremental build from shipping a stale package that installs as the already-installed version.
