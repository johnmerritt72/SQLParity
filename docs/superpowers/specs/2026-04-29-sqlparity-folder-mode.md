# SQLParity — Folder Mode (Side B as Solution Folder)

**Date:** 2026-04-29
**Status:** Spec — pending implementation
**Targets:** v1.2

## 1. Summary

Allow Side B of a comparison to point at a folder of `.sql` files (the user's open SSMS Solution) instead of a live database. The folder is treated as a *virtual database*: SQLParity parses each `.sql` file, extracts CREATE statements, and produces a `DatabaseSchema` with the same shape as a SMO-read schema. Comparison and results UI are unchanged from the user's perspective. The new sync direction A→B writes per-object `.sql` files into the solution folder.

### Motivation

Teams keep a subset of database objects (typically procs, views, functions, sometimes tables) under source control as `.sql` files inside a SSMS solution. Today, comparing the live DB to those files is a manual chore. Folder mode collapses that chore into the same drift-detection workflow already used for DB-to-DB compares.

## 2. Non-goals (v1.2)

- Subfolder recursion (root only). Designed so recursion can be added later without redesign.
- Migration script generation when a table's columns change. Files written for tables are full `CREATE TABLE` definitions wrapped in `IF NOT EXISTS`; transforming an existing table is out of scope and explicitly the user's responsibility for now.
- Column-level structured diff for tables in folder mode. Tables are compared by DDL text only. The full column-tree / risk-tier flow only fires for DB↔DB compares in v1.2. Files we write for tables are still well-formed CREATE statements; we just don't decompose CREATE-TABLE text on the way *in*.
- Custom file-naming format. Hard-coded `<TYPE>.<schema>.<name>.sql` for v1.2; user-configurable format is v1.3.

## 3. User-visible behavior

### 3.1 Side B mode toggle
The Side B connection panel gains a Mode selector: **Database** (today's behavior) or **Folder**. Side A always remains a Database. The Folder option is disabled with a tooltip ("Open a Solution in Solution Explorer to enable folder mode") whenever no SSMS solution is loaded.

### 3.2 Folder source
When Mode = Folder, the panel reads the open solution's directory from the SSMS DTE (`DTE.Solution.FullName`). The user does not browse — folder = current solution. (Multi-project solutions: v1.2 uses the *solution* directory; if needs evolve we can add a project picker.)

### 3.3 Comparison filtering
New comparison option **Limit Comparison to Solution Folder Objects** (default ON). When ON: any change whose object Id is missing from the folder side is suppressed. Empty folder is treated as an automatic no-op for the limit (i.e. show all of A — explicit user requirement).

### 3.4 Sync direction A→B
Live-apply button label becomes **"Write to Folder"**. Behavior per change:

- **Modified** + object lives in a single-object file → overwrite that file with A's DDL, keep filename.
- **Modified** + object lives in a multi-object file → split: write a new individual file *for every* object that file currently defines, then comment-out the entire original file with `/* ... */` and write a `-- Replaced by: <list>` trailer above the comment block. (We split *all* objects in the file, not just the one that changed, per explicit user requirement: "force each object to be in its own sql file.")
- **New on A / not in B** → create a new file using the naming convention.
- **Dropped on A / file-only object** → comment out the file's CREATE block and append a guarded DROP wrapper:
  ```sql
  IF OBJECT_ID(N'[dbo].[Foo]', N'U') IS NOT NULL
      DROP TABLE [dbo].[Foo];
  ```
  The original CREATE stays in the file as a `/* ... */` block (historical reference per user request).

After writes, modified/added files are registered with the open SSMS solution via the DTE so they appear in Solution Explorer immediately.

### 3.5 Sync direction B→A
Reuses the existing `ScriptGenerator` / `LiveApplier`. The `Change` records produced by the comparator already carry both DDL strings; they don't care that Side B came from a file. No new code on this path beyond the comparator wiring.

### 3.6 New options summary
| Option | Default | Effect |
| --- | --- | --- |
| Limit Comparison to Solution Folder Objects | ON | Hide A-only changes when B is folder mode. |
| Record Sync History | ON | Today's HistoryWriter behavior. OFF disables history writes for DB syncs. Folder syncs never write history regardless. |

## 4. File templates

### 4.1 Header for every generated file

```sql
USE [<DatabaseNameAtSyncTime>];
GO

-- Generated by SQLParity from <ServerName>/<DatabaseName> on <YYYY-MM-DD HH:mm UTC>.
-- Object: <TYPE> <schema>.<name>
-- Re-running this script is safe — see the IF (NOT) EXISTS guard below.
GO

<the original DDL (which may contain its own header comments) — NOT prefixed
with anything from us>
GO
```

Two `GO`s are non-negotiable:

1. **After `USE`** — standard T-SQL.
2. **After the SQLParity header comment, before the CREATE batch.** This is critical for module-type objects (PROC, FUNCTION, VIEW, TRIGGER): SQL Server stores the *entire* batch text submitted with a CREATE in `sys.sql_modules.definition`. Without the GO, our "Generated by SQLParity" lines would be persisted as part of the deployed proc and would re-appear above CREATE on every subsequent script-out — polluting downstream comparisons. The GO isolates our prelude into its own batch so only the original DB-side header comments (Author, Description, etc.) end up in `sql_modules`.

By the same rule, **any header comments that already exist above CREATE on the DB side must be preserved verbatim** when we write the file. SMO's scripting output already includes them; the writer must not strip, re-format, or re-order them.

The `USE` line is always emitted, even though it's redundant if the user already has the right DB selected. It makes the script copy-paste-safe into a fresh SSMS query window.

### 4.2 CREATE-OR-ALTER types (Procedures, Functions, Views, Triggers)

```sql
CREATE OR ALTER PROCEDURE [dbo].[GetOrders] @id INT
AS
BEGIN
    SELECT ...
END
GO
```

`CREATE OR ALTER` requires SQL Server 2016 SP1+. Per user direction, "always use CREATE OR ALTER when possible." Files generated this way are idempotent and re-runnable.

### 4.3 IF-NOT-EXISTS types (Tables, UDDTs, Table Types, Sequences, Schemas, Synonyms)

User explicitly chose **skip-if-exists** over **drop-and-recreate** for all non-OR-ALTER types: *"I really really don't want someone to run a script and drop the table."* Synonyms get the same treatment for consistency, even though dropping them is cheap.

#### Table
```sql
IF OBJECT_ID(N'[dbo].[Orders]', N'U') IS NULL
BEGIN
    CREATE TABLE [dbo].[Orders] (
        Id INT NOT NULL,
        ...
    );
END
GO
```

#### Schema (CREATE SCHEMA must be the first statement in its batch — wrap in EXEC)
```sql
IF NOT EXISTS (SELECT 1 FROM sys.schemas WHERE name = N'audit')
    EXEC(N'CREATE SCHEMA [audit]');
GO
```

#### User-Defined Data Type
```sql
IF NOT EXISTS (
    SELECT 1 FROM sys.types
    WHERE schema_id = SCHEMA_ID(N'dbo') AND name = N'OrderId'
)
    CREATE TYPE [dbo].[OrderId] FROM INT NOT NULL;
GO
```

#### Table Type
```sql
IF NOT EXISTS (
    SELECT 1 FROM sys.types
    WHERE schema_id = SCHEMA_ID(N'dbo') AND name = N'OrderRows' AND is_table_type = 1
)
BEGIN
    CREATE TYPE [dbo].[OrderRows] AS TABLE (...);
END
GO
```

#### Sequence
```sql
IF NOT EXISTS (
    SELECT 1 FROM sys.sequences
    WHERE schema_id = SCHEMA_ID(N'dbo') AND name = N'OrderSeq'
)
BEGIN
    CREATE SEQUENCE [dbo].[OrderSeq] AS BIGINT START WITH 1 INCREMENT BY 1;
END
GO
```

#### Synonym
```sql
IF NOT EXISTS (
    SELECT 1 FROM sys.synonyms
    WHERE schema_id = SCHEMA_ID(N'dbo') AND name = N'LegacyOrders'
)
    CREATE SYNONYM [dbo].[LegacyOrders] FOR [dbo].[Orders];
GO
```

### 4.4 Naming convention
`<TYPE>.<schema>.<name>.sql`, lowercase preserved from the source DDL. Type prefixes:

| Object | Prefix |
| --- | --- |
| Stored procedure | `PROC` |
| User-defined function | `FUNC` |
| View | `VIEW` |
| Table | `TABLE` |
| User-defined data type | `TYPE` |
| Table type | `TYPE` |
| Sequence | `SEQ` |
| Synonym | `SYN` |
| Trigger | `TRIG` |
| Schema | `SCHEMA` (no `<name>` part) |

Example: `PROC.dbo.GetOrders.sql`. Conflict policy: if a file with the target name already exists and contains a *different* object, append `_1`, `_2`, ... until unique. (Existing files holding *this* object keep their name and are overwritten.)

### 4.5 Encoding
UTF-8 with BOM. Matches what SSMS produces by default and avoids ANSI-codepage character corruption on file edits.

## 5. Components

### 5.1 New: `SqlFileParser` (in `SQLParity.Core`)

Pure function `Parse(string sqlText) -> IReadOnlyList<ParsedObject>`.

A `ParsedObject` carries:
- `ObjectType` (Procedure / Function / View / Table / Trigger / Type / Sequence / Synonym / Schema)
- `SchemaQualifiedName` (defaulting schema to `dbo` when unqualified)
- `string Ddl` — the raw text of the object's batch, preserving original casing/whitespace
- `int BatchIndex` — for diagnostics and round-tripping

Tokenizer respects `'string'`, `[bracket]`, `--`, `/* */`, and case-insensitive `GO` (must be on its own line, possibly with whitespace). For each batch, scans for the first non-comment/non-string token and matches `CREATE [OR ALTER]? (PROC|PROCEDURE|FUNCTION|VIEW|TABLE|TRIGGER|TYPE|SEQUENCE|SYNONYM|SCHEMA)` followed by an identifier (bracketed or not, optionally schema-qualified). Batches without a recognized CREATE are silently skipped (allows files to contain `USE`, comments, etc. without errors).

### 5.2 New: `FolderSchemaReader` (in `SQLParity.Core`)

`ReadSchema(string folderPath) -> (DatabaseSchema, FolderSchemaContext)` where `FolderSchemaContext` carries:
- `Dictionary<SchemaQualifiedName, FileBacking> ObjectToFile` — for each parsed object, the source `.sql` path plus a flag for whether that file holds *only* that object.
- `IReadOnlyList<string> ParseWarnings` — surfaces "couldn't determine object type for batch N in foo.sql" diagnostics.

Reader does NOT recurse into subfolders in v1.2; signature parameter `bool recursive` defaults to false and is wired but unused (avoids a future API break).

### 5.3 New: `IdempotentDdlWrapper` (in `SQLParity.Core`)

Pure function: given an object type, schema, name, and bare CREATE text, returns the wrapped re-runnable form per the templates in §4.2 / §4.3. Stripped of any `USE`/`GO`/header — those are added by the writer.

### 5.4 New: `FolderSyncWriter` (in `SQLParity.Core`)

Drives A→B sync writes. Inputs: a list of `Change`s, the `FolderSchemaContext` from the read, the source DB name (for the `USE` line). Behaviors:

1. Group changes by source file (where applicable).
2. Files identified as "multi-object" get split: emit individual files for each contained object, then write the original file's content wrapped in `/* ... */`.
3. Single-object files get overwritten in place.
4. New objects get new files with the naming convention.
5. Returns a manifest: `(FilesWritten, FilesCommentedOut, FilesCreated, Skipped)` for the SE-registration step.

### 5.5 VSIX shell additions

- `ConnectionSideViewModel` gains a `Mode` enum and folder-mode property. Side A's view-model never exposes Folder.
- `ConnectionSetupView.xaml` gets a Mode toggle for Side B's panel.
- A small `SsmsSolutionService` helper wraps `DTE` calls: `IsSolutionOpen`, `GetSolutionDirectory()`, `RegisterFile(string path)`, `CommentedOutFiles(IEnumerable<string>)`. Lives in `SQLParity.Vsix` since `SQLParity.Core` cannot reference VS interop assemblies.
- `ComparisonHostViewModel` reads new options and passes through to the comparator and writer.

## 6. Comparator changes

`SchemaComparator.Compare` gains:

```csharp
bool limitToFolderObjects = false   // suppresses A-only changes when true
```

Only meaningful when Side B is folder-sourced; in DB-to-DB compares the caller passes `false`. Empty-folder behavior is implemented at the call site (host VM): if Side B's schema is empty, force `limitToFolderObjects = false` so the user sees A's full inventory.

The "limit" filter applies *after* the existing change set is built — strictly subtractive — so risk classification, pre-flight queries, and the gauntlet logic remain identical.

## 7. Open questions deferred to v1.3+

- **Migration scripts for table column changes.** v1.2 writes `IF NOT EXISTS` + full CREATE TABLE; if the table already exists with different columns, the script does nothing. Future work: a separate "Generate Migration" output that emits `ALTER TABLE` blocks. Will require its own UI affordance to make clear it's destructive-by-intent.
- **Subfolder recursion.** Reader signature already accepts a `recursive` flag.
- **Custom naming formats.** A template string option (e.g. `{type}.{schema}.{name}.sql`) is the obvious shape; design when needed.
- **Multi-project SSMS solutions.** v1.2 uses the solution root. If a user has multiple projects under one solution, we'll need a per-comparison "active project" picker.

## 8. Test plan

- `SqlFileParser` unit tests: GO splits across edge cases (case, leading whitespace, embedded "GO" inside strings/comments), `CREATE OR ALTER` recognition, default-to-dbo schema resolution, bracketed identifiers with special chars, files with no CREATE statements, files mixing multiple object types, comment-with-fake-CREATE.
- `FolderSchemaReader` unit tests: empty folder, single-object files, multi-object files, junk file present, `FolderSchemaContext.ObjectToFile` correctness, single-vs-multi flag.
- `IdempotentDdlWrapper` unit tests: each object type's template, schema name with bracket-required casing, EXEC-wrapped CREATE SCHEMA.
- `SchemaComparator` unit tests: `limitToFolderObjects` on/off, empty Side B + limit-on shows everything from A, only-on-B object surfaces a Dropped change correctly.
- `FolderSyncWriter` unit tests: round-trip a single-object file overwrite, multi-object split + comment-out, new file naming + collision, dropped object writes guarded DROP plus commented-out original.
- VSIX-shell integration: visually confirm Mode toggle, SE disable when no solution, files appear in SE after sync.
